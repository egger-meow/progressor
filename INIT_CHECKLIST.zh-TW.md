# 初始化清單

順序在這裡很重要:每一步都需要前面幾步是真的(不是模板佔位符),才能誠實地
填寫。如果還沒讀過,先讀
[`LOOP_ENGINEERING.zh-TW.md`](LOOP_ENGINEERING.zh-TW.md)——這份清單只是機械
式的執行順序;那份文件解釋了為什麼順序要這樣排。

**不想手動走完這份清單?** [`BOOTSTRAP.zh-TW.md`](BOOTSTRAP.zh-TW.md) 把同
一份清單變成 agent 主導的訪談:你貼上專案想法、一批回答完問題、讀一份簡短的
授權摘要——其他全部由 agent 打字。無論走哪條路,這份清單都仍然是規格;
bootstrap 是一步一步照著它走,不是繞過它。

跟擁有這個專案的人類一起走完這份清單——步驟 1-4 需要他們的判斷,不是 agent
的猜測。Agent 可以起草,但要由人類確認過後,才能被當作已授權。

過程中把 [`examples/linkcheck/`](examples/linkcheck/) 開在另一個分頁——它是
這份清單要你寫的每一個檔案的完整填寫實例,讓你有一個具體的模型可以參考,而不
只是抽象的模板。（這個範例本身只有英文版,沒有繁中版——它是「填好長什麼樣
子」的示範,不是要逐字翻譯的說明文件。）

**在第 1 步之前:** 如果你把 loop-engine 自己的 `README.md`(或
`README.zh-TW.md`)複製進了你的專案,把它們換成你專案自己的 README——它們
描述的是 loop-engine,不是你的專案,而且不像下面的檔案那樣需要被模板化。
`INBOX.md` 是相反的情況:它開箱即用,沒有東西要填——保持原樣(而且是空的)
就好。`FRAMEWORK_FEEDBACK.md` 也一樣——在框架本身給你可回報的東西之前,
保持空著。

- [ ] **1. `docs/project-charter.md`** —— 使命、核心領域、護欄。這是全專案
      權威最高的文件;其他一切都應該能回溯到它。如果你現在還沒辦法有信心地填
      好這份文件,代表這個專案還沒準備好進行自主迴圈——花更多時間在這裡,而
      不是更少。
- [ ] **2. `docs/domain-model.md`** —— 為 charter 的「Core Areas」裡的核心
      概念命名。在詳細寫任何其他文件之前先做這一步,讓後面的文件從一開始就用
      一致的詞彙,不用之後再回頭改名。
- [ ] **3. `docs/system-direction.md`** —— 目標架構,細節要足夠指導真實的決
      策。全新專案可以寫得簡略(還沒有「現況落差」這回事);如果是要把這套框
      架套進既有程式碼庫,就該寫得扎實一點。
- [ ] **4. `ROADMAP.md`** —— 預先授權的 phase 佇列。至少在「Authorized
      Phases」下寫出第一個 phase(名稱、目標、退場條件);更大的想法先放進
      「Proposed — Not Yet Authorized」。退場條件是大家最容易寫得太模糊的一
      步——「X 運作良好」永遠收不了尾;要寫得像驗收關卡一樣具體。這裡是人類
      的授權真正被花掉的地方,所以跟 charter 一樣,是人類的決定,agent 可以
      幫忙起草。
- [ ] **5. `docs/status.md`** —— 對全新專案來說,這份文件一開始幾乎是空的,
      只需定義**task gate** 指令;等第一個 phase 的退場條件確定要證明什麼之
      後,再加上**phase gate**(收尾一個 phase 時要跑的、昂貴的端到端檢查)。
      如果是既有程式碼庫,這裡也是你誠實、詳細記錄「現在到底有什麼真的能動」
      的地方——現在寫得越準確,agent 之後就越不用浪費時間重新摸索。
- [ ] **6. `docs/build-status.md`** —— 第 5 步的粗粒度表格版本,每個 charter
      裡的核心領域一列。用今天的日期和已經驗證過的東西,種下 Verification
      Evidence 日誌的第一筆。
- [ ] **7. `docs/release.md`** —— 版本規則與發佈清單。還沒發佈過的專案可以寫
      得很精簡;還是值得先有這個形狀,免得日後在時間壓力下才臨時要用。
- [ ] **8. `CLAUDE.md` 和 `AGENTS.md`** —— 填入指令、結構、慣例。這兩份文件
      應該連結回上面那些文件,而不是把內容複製進來——忍住把內容抄進來的衝
      動。
- [ ] **9. `PRIORITIES.md`** —— 先寫這個專案的「What Counts as a Blocker」
      (這是大家最容易想跳過或寫得很籠統的一步——別這樣;模糊的 blocker 定
      義,會產生一個 agent 沒辦法可靠推理的優先佇列)。然後啟用 `ROADMAP.md`
      的第一個 phase,把它拆解進「Current Priorities」——真實的項目,最急迫
      的排最前面,每一項都要寫清楚「done means」是什麼。
- [ ] **10. 刪掉所有殘留的 `TEMPLATE:` 註解。** 執行
      `./scripts/check-templates.sh`(Windows 上用
      `check-templates.ps1`),不要用肉眼一個一個找——它會列出每一個還帶著
      `TEMPLATE:` 標記的檔案和行號,有殘留就非零結束。一份文件裡還留著
      `TEMPLATE:` 註解,就代表它還不是正典來源——在註解消失之前,把它當成
      「還沒寫」。(`docs/audits/TEMPLATE.md` 本身故意被排除在掃描之外——它
      本來就該永遠是一份空白模板;見 `docs/audits/README.md`。`BOOTSTRAP.md`
      和它的 `.zh-TW` 版本也被排除,因為它們以指令的形式逐字引用了
      bootstrap 授權標記。)
- [ ] **11. 真的跑一次迴圈。** 挑 `PRIORITIES.md` 的第一項,走完
      `LOOP_ENGINEERING.zh-TW.md` 裡完整的 task loop——實作、用
      `docs/status.md` 的 task gate 驗證、更新現況文件、移除該優先項目、記
      錄 changelog 條目。過程中在 `INBOX.md` 丟一句小備註,確認 agent 有照協
      議翻譯並清空它。這一步會在成本還低的時候,浮現出上面文件的任何缺口(缺
      少的脈絡、沒定義的關卡、模糊的 blocker 定義)——好過等到真的要依賴這套
      框架做無人監督的工作時才發現。

第 11 步做完之後,一個在這個 repo 裡工作的 agent,應該能開一個全新的
session、讀 `AGENTS.md`/`CLAUDE.md`,就正確判斷出接下來該做什麼,不需要你在
這次對話裡重新解釋任何東西——而且你也應該能透過寫 `INBOX.md` 中途轉向它,而
不用在聊天裡打斷它。

（本檔案是 [`INIT_CHECKLIST.md`](INIT_CHECKLIST.md) 的繁體中文版，兩者應保持
內容一致；若有出入，以英文版為準。）
